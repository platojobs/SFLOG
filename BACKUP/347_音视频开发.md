## 第一部分：面试题库（按模块分类）

### 1. 网络协议篇 (RTSP/RTP/RTCP/TCP/UDP)

#### 基础题

**1. RTSP 与 HTTP 的区别是什么？**

**答案：**
- **状态管理**：RTSP 是有状态（stateful）协议，维护会话状态；HTTP 是无状态（stateless）协议。
- **控制方式**：RTSP 控制流与数据流分离，可以独立控制媒体流的播放、暂停等操作。
- **传输层**：RTSP 通常基于 TCP，但也可以使用 UDP；HTTP 基于 TCP。
- **方法**：RTSP 有特有的方法如 SETUP、PLAY、PAUSE、TEARDOWN 等；HTTP 有 GET、POST 等方法。
- **默认端口**：RTSP 默认端口 554；HTTP 默认端口 80。

**2. 请简述 RTSP 的基本交互流程。**

**答案：**
RTSP 的基本交互流程如下：

1. **OPTIONS**：客户端向服务器查询可用的方法。
2. **DESCRIBE**：客户端请求媒体会话的描述信息，通常以 SDP 格式返回。
3. **SETUP**：客户端与服务器协商传输参数，如传输协议（TCP/UDP）、端口等。
4. **PLAY**：客户端请求开始播放媒体流。
5. **PAUSE**：客户端请求暂停播放。
6. **TEARDOWN**：客户端请求结束会话，释放资源。

**3. SDP 协议的作用是什么？**

**答案：**
SDP（Session Description Protocol）是会话描述协议，用于描述媒体会话的信息，包括：
- 媒体类型（音频/视频）
- 编解码器信息
- 网络传输信息（IP地址、端口）
- 时间信息
- 会话名称和描述

SDP 通常在 RTSP DESCRIBE 请求的响应中返回，帮助客户端了解如何接收和处理媒体流。

**4. RTP 和 RTCP 的作用分别是什么？**

**答案：**
- **RTP（Real-time Transport Protocol）**：负责传输实时数据，如音频、视频等媒体数据。RTP 提供了时间戳、序列号等机制，帮助接收端进行同步和重排序。
- **RTCP（Real-time Transport Control Protocol）**：负责传输质量控制信息，如：
  - 丢包统计
  - 延迟和抖动信息
  - 网络状况反馈
  - 会话成员管理

RTCP 通常与 RTP 一起使用，提供端到端的实时通信质量监控。

**5. 简述 TCP 三次握手和四次挥手的过程。**

**答案：**

**TCP 三次握手：**
1. **SYN**：客户端发送 SYN 包（seq = x），进入 SYN_SENT 状态。
2. **SYN-ACK**：服务器收到 SYN 包，发送 SYN-ACK 包（seq = y, ack = x+1），进入 SYN_RECV 状态。
3. **ACK**：客户端收到 SYN-ACK 包，发送 ACK 包（ack = y+1），进入 ESTABLISHED 状态。服务器收到 ACK 包后也进入 ESTABLISHED 状态。

**TCP 四次挥手：**
1. **FIN**：主动关闭方发送 FIN 包，进入 FIN_WAIT_1 状态。
2. **ACK**：被动关闭方收到 FIN 包，发送 ACK 包，进入 CLOSE_WAIT 状态。
3. **FIN**：被动关闭方完成数据发送后，发送 FIN 包，进入 LAST_ACK 状态。
4. **ACK**：主动关闭方收到 FIN 包，发送 ACK 包，进入 TIME_WAIT 状态，等待 2MSL 后关闭连接。

**为什么挥手需要四次？**
- 因为 TCP 是全双工通信，每个方向都需要单独关闭。
- 被动关闭方在收到 FIN 后，可能还有数据要发送，所以先发送 ACK，等数据发送完毕后再发送 FIN。

**TIME_WAIT 状态的作用：**
- 确保最后一个 ACK 包能够到达对方，避免对方重发 FIN 包。
- 等待网络中可能存在的旧连接的数据包过期，避免对新连接造成干扰。

#### 进阶题

**6. RTSP 中，SETUP 阶段协商了哪些关键信息？**

**答案：**
SETUP 阶段协商的关键信息包括：
- 传输协议（TCP/UDP）
- 客户端接收端口
- 服务器发送端口
- 组播信息（如果使用组播）
- 传输模式（单播/组播）
- 会话标识符
- 媒体流的控制 URI

**7. RTP 包头的结构是怎样的？**

**答案：**
RTP 包头结构（12字节）如下：

| 字段 | 长度 | 描述 |
|------|------|------|
| 版本（V） | 2位 | RTP 版本号 |
| 填充位（P） | 1位 | 指示包末尾是否有填充 |
| 扩展位（X） | 1位 | 指示是否有扩展头 |
| CSRC计数（CC） | 4位 | 贡献源标识符数量 |
| 标记位（M） | 1位 | 用于标记帧边界 |
| 载荷类型（PT） | 7位 | 指示载荷的类型（如 H.264、AAC） |
| 序列号（SN） | 16位 | 包的序列号，用于重排序 |
| 时间戳（TS） | 32位 | 媒体数据的时间戳 |
| SSRC | 32位 | 同步源标识符 |
| CSRC | 0-15个32位 | 贡献源标识符列表 |

**8. 时间戳在 RTP 中的重要性是什么？它是如何工作的？**

**答案：**
**重要性：**
- 用于接收端进行播放同步（音视频同步）
- 用于抖动消除
- 用于丢包检测

**工作原理：**
- 时间戳通常基于一个线性增长的时钟（如 90kHz 时钟用于视频，48kHz 时钟用于音频）
- 每帧数据对应一个时间戳
- 接收端根据时间戳计算出正确的播放时间，确保媒体流畅播放

**9. 当使用 UDP 传输 RTP 包时，如何处理丢包和乱序？**

**答案：**
**处理丢包：**
- **视频**：使用错误隐藏技术（如帧内预测、运动补偿）掩盖丢包造成的质量问题
- **音频**：使用 PLC（Packet Loss Concealment）技术，根据前后音频数据插值模拟丢失的音频
- **关键帧请求**：当丢失关键帧时，通过 RTCP FIR（Full Intra Request）或 PLI（Picture Loss Indication）请求服务器重新发送关键帧

**处理乱序：**
- 使用 RTP 序列号进行重排序
- 实现抖动缓冲区（Jitter Buffer），存储一定数量的包，根据时间戳和序列号进行排序后再播放

**10. RTCP 的 SR 和 RR 包分别包含哪些关键信息？**

**答案：**

**SR（Sender Report）包：**
- NTP 时间戳（网络时间协议时间戳）
- RTP 时间戳（与 NTP 时间戳对应）
- 发送的包数
- 发送的字节数
- 每个接收者的 RR 信息

**RR（Receiver Report）包：**
- 丢包率
- 累计丢包数
- 最高序列号
- 到达抖动
- 时延（从发送到接收的时间）

**11. RTSP over TCP 的实现原理是什么？为什么有时候需要这样做？**

**答案：**
**实现原理：**
- 将 RTP/RTCP 数据通过 RTSP 的同一个 TCP 连接发送
- 使用特殊的封装格式，在 TCP 流中区分 RTSP 信令和 RTP/RTCP 数据

**为什么需要：**
- 解决 UDP 穿透 NAT/防火墙的问题
- 在网络环境复杂（如企业内网）时提高连接成功率
- 避免 UDP 端口被封锁的问题

**缺点：** 增加延迟，因为 TCP 的可靠性机制会导致数据传输延迟增加。

### 2. 编解码与封装篇 (FFmpeg/H.264/AAC/容器格式)

#### 基础题

**1. 什么是 I 帧、P 帧、B 帧？它们的区别和关系是什么？**

**答案：**
- **I 帧（Intra-coded Frame）**：关键帧，包含完整的图像信息，不依赖其他帧。
- **P 帧（Predictive Frame）**：前向预测帧，只包含与前一帧的差异信息，依赖前一帧。
- **B 帧（Bi-directional Frame）**：双向预测帧，包含与前后帧的差异信息，依赖前后帧。

**关系：**
- I 帧是压缩的基础，P 帧和 B 帧通过参考其他帧来减少数据量
- 解码顺序：I → P → B（因为 B 帧需要参考后面的 P 帧）
- 显示顺序：I → B → P

**2. H.264 的码流结构是怎样的？NAL 单元是什么？**

**答案：**
**H.264 码流结构：**
- 由一系列 NAL（Network Abstraction Layer）单元组成
- 每个 NAL 单元包含一个头部和载荷数据

**NAL 单元：**
- NAL 是网络抽象层，用于封装编码后的数据
- NAL 单元类型包括：
  - SPS（Sequence Parameter Set）：序列参数集，包含分辨率、帧率等
  - PPS（Picture Parameter Set）：图像参数集，包含熵编码类型等
  - IDR（Instantaneous Decoding Refresh）：即时解码刷新，相当于关键帧
  - SLICE：图像切片数据

**3. 常见的音频编码格式有哪些？AAC 编码的基本原理是什么？**

**答案：**
**常见音频编码格式：**
- AAC（Advanced Audio Coding）
- MP3（MPEG-1 Audio Layer III）
- Opus
- Vorbis
- FLAC（无损）

**AAC 编码基本原理：**
1. **时域到频域转换**：使用 MDCT（Modified Discrete Cosine Transform）将时域信号转换为频域
2. **听觉掩蔽效应**：利用人耳对不同频率声音的敏感度不同，忽略人耳不易察觉的声音
3. **量化**：对频域系数进行量化，减少数据量
4. **熵编码**：使用霍夫曼编码或算术编码进一步压缩数据

**4. FFmpeg 的解码流程是怎样的？**

**答案：**
FFmpeg 解码流程如下：

1. **打开输入文件**：`avformat_open_input`
2. **获取流信息**：`avformat_find_stream_info`
3. **找到解码器**：`avcodec_find_decoder`
4. **打开解码器**：`avcodec_open2`
5. **读取数据包**：`av_read_frame`
6. **发送数据包到解码器**：`avcodec_send_packet`
7. **接收解码后的帧**：`avcodec_receive_frame`
8. **处理解码后的帧**（如渲染）
9. **释放资源**：`avcodec_close`、`avformat_close_input`

#### 进阶题

**5. 什么是 DTS 和 PTS？为什么在视频编码中它们会不同？**

**答案：**
- **DTS（Decoding Time Stamp）**：解码时间戳，指示帧应该何时被解码
- **PTS（Presentation Time Stamp）**：显示时间戳，指示帧应该何时被显示

**为什么不同：**
因为 B 帧的存在，解码顺序和显示顺序不一致。B 帧需要参考后面的 P 帧，所以必须先解码后面的 P 帧，再解码 B 帧。但显示时，B 帧应该在 P 帧之前显示。

**示例：**
- 编码顺序：I1 → P3 → B2
- 解码顺序：I1 → P3 → B2
- 显示顺序：I1 → B2 → P3

**6. 如何从 H.264 码流中提取 SPS 和 PPS 信息？它们包含哪些关键参数？**

**答案：**
**提取方法：**
1. 解析 H.264 码流中的 NAL 单元
2. 根据 NAL 单元类型（SPS：7，PPS：8）识别 SPS 和 PPS 单元
3. 解析 SPS 和 PPS 的具体内容

**SPS 包含的关键参数：**
- 分辨率（width, height）
- 帧率
- Profile 和 Level
- 色彩空间
- 熵编码模式

**PPS 包含的关键参数：**
- 熵编码类型
- 量化参数初始值
- 最大参考帧数

**7. 软解和硬解的区别是什么？在 iOS 上如何实现硬解？**

**答案：**
**软解和硬解的区别：**
| 特性 | 软解 | 硬解 |
|------|------|------|
| 处理单元 | CPU | GPU/专用硬件 |
| 兼容性 | 高 | 依赖硬件支持 |
| 功耗 | 高 | 低 |
| 性能 | 较低 | 高 |
| 灵活性 | 高 | 较低 |

**iOS 上实现硬解：**
使用 VideoToolbox 框架：
1. 创建解码器会话：`VTDecompressionSessionCreate`
2. 配置解码器参数
3. 发送压缩数据：`VTDecompressionSessionDecodeFrame`
4. 接收解码后的 CVPixelBuffer
5. 释放会话：`VTDecompressionSessionInvalidate`

**8. FFmpeg 中 `avcodec_send_packet` 和 `avcodec_receive_frame` 这种异步API的好处是什么？**

**答案：**
- **解耦输入输出**：可以先发送多个数据包，然后再接收解码后的帧，提高处理效率
- **更灵活的错误处理**：可以分别处理输入和输出的错误
- **支持多线程解码**：利用硬件和软件的并行处理能力
- **更好的内存管理**：可以更灵活地控制缓冲区大小
- **适应不同的解码速度**：解码速度慢时不会阻塞输入线程

### 3. 渲染与显示篇 (OpenGL ES/YUV)

#### 基础题

**1. 什么是 YUV 颜色空间？它与 RGB 的区别是什么？**

**答案：**
**YUV 颜色空间：**
- Y：亮度（Luminance）
- U：色度（Chrominance），蓝色分量与亮度的差值
- V：色度（Chrominance），红色分量与亮度的差值

**与 RGB 的区别：**
- **存储方式**：RGB 每个像素需要 3 个分量，YUV 可以通过减少色度采样来压缩数据
- **人眼敏感度**：人眼对亮度更敏感，对色度不太敏感，所以 YUV 更适合压缩
- **传输效率**：YUV 数据量更小，传输更高效
- **应用场景**：YUV 广泛用于视频编码和传输，RGB 用于显示

**2. 常见的 YUV 采样格式有哪些？**

**答案：**
- **YUV444**：每个像素都有完整的 Y、U、V 分量
- **YUV422**：每 2 个像素共享一组 U、V 分量
- **YUV420**：每 4 个像素共享一组 U、V 分量

**播放器中最常用的格式：**
- **YUV420P**： planar 格式，Y、U、V 三个平面分开存储
- **NV12**： planar 格式，Y 平面单独存储，UV 交错存储

**3. OpenGL ES 渲染 YUV 数据的原理是什么？**

**答案：**
OpenGL ES 渲染 YUV 数据的原理是使用片段着色器将 YUV 转换为 RGB：

1. **创建纹理**：为 Y、U、V 分量创建纹理（YUV420P 需要 3 个纹理，NV12 需要 2 个纹理）
2. **填充纹理数据**：将 YUV 数据上传到对应的纹理
3. **顶点着色器**：传递纹理坐标
4. **片段着色器**：采样纹理，将 YUV 转换为 RGB
5. **渲染**：将转换后的 RGB 数据渲染到屏幕

**YUV 转 RGB 的公式：**
```
R = Y + 1.402 * (V - 128)
G = Y - 0.344 * (U - 128) - 0.714 * (V - 128)
B = Y + 1.772 * (U - 128)
```

**4. EGL 的作用是什么？**

**答案：**
EGL（Embedded System Graphics Library）是 OpenGL ES 与窗口系统之间的桥梁，主要作用：
- 创建和管理渲染上下文（EGLContext）
- 创建和管理绘制表面（EGLSurface）
- 处理 OpenGL ES 与本地窗口系统的通信
- 管理颜色缓冲区和深度缓冲区

#### 进阶题

**5. YUV420P 和 NV12 在内存布局上有何不同？在 OpenGL ES 中如何分别处理？**

**答案：**

**内存布局：**
- **YUV420P**：planar 格式，三个平面分开存储
  ```
  Y Y Y Y
  Y Y Y Y
  U U
  V V
  ```
- **NV12**：planar 格式，Y 平面单独存储，UV 交错存储
  ```
  Y Y Y Y
  Y Y Y Y
  U V U V
  ```

**OpenGL ES 处理：**
- **YUV420P**：需要 3 个采样器，分别采样 Y、U、V 纹理
- **NV12**：需要 2 个采样器，一个采样 Y 纹理，一个采样 UV 纹理（UV 交错数据在着色器中分离）

**6. 如何在 iOS 上高效地将解码后的 YUV 数据传递给 OpenGL ES？**

**答案：**
使用 CVOpenGLESTextureCache 实现零拷贝渲染：

1. **创建纹理缓存**：`CVOpenGLESTextureCacheCreate`
2. **从 CVPixelBuffer 创建纹理**：`CVOpenGLESTextureCacheCreateTextureFromImage`
3. **绑定纹理**：`glBindTexture`
4. **渲染**：在着色器中使用纹理
5. **释放纹理**：纹理会随着 CVPixelBuffer 的释放而自动释放

**优点：** 避免了内存拷贝，提高了渲染效率。

**7. OpenGL 渲染时，如何解决图像拉伸的问题？**

**答案：**
解决图像拉伸问题的方法：

1. **计算宽高比**：获取纹理的宽高比和视图的宽高比
2. **调整视口**：根据宽高比调整视口大小，保持图像比例
3. **调整纹理坐标**：根据宽高比调整纹理坐标，裁剪或填充图像
4. **使用 letterbox/pillarbox**：在屏幕两侧添加黑边，保持图像原始比例

**代码示例：**
```objc
// 计算宽高比
CGFloat videoAspectRatio = (CGFloat)videoWidth / videoHeight;
CGFloat viewAspectRatio = (CGFloat)viewWidth / viewHeight;

if (videoAspectRatio > viewAspectRatio) {
    // 视频比视图宽，左右裁剪
    CGFloat scale = viewHeight / videoHeight;
    CGFloat scaledWidth = videoWidth * scale;
    CGFloat xOffset = (scaledWidth - viewWidth) / 2.0;
    // 调整纹理坐标
} else {
    // 视频比视图高，上下裁剪
    CGFloat scale = viewWidth / videoWidth;
    CGFloat scaledHeight = videoHeight * scale;
    CGFloat yOffset = (scaledHeight - viewHeight) / 2.0;
    // 调整纹理坐标
}
```

**8. 介绍一下 OpenGL ES 的渲染管线。**

**答案：**
OpenGL ES 渲染管线包括以下阶段：

1. **顶点着色器**：处理顶点数据，进行坐标变换、光照计算等
2. **图元装配**：将顶点组装成图元（点、线、三角形）

### 4. 多线程与同步篇 (POSIX Thread/音视频同步)

#### 基础题

**1. 线程同步的方式有哪些？**

**答案：**
线程同步的方式包括：
- **互斥锁（pthread_mutex_t）**：保护临界区，同一时间只有一个线程可以访问
- **条件变量（pthread_cond_t）**：用于线程间的信号通知
- **信号量（sem_t）**：控制对资源的访问数量
- **读写锁**：允许多个线程同时读，只有一个线程可以写
- **原子操作**：使用 CPU 指令保证操作的原子性

**2. 什么是生产者-消费者问题？如何用条件变量实现？**

**答案：**
**生产者-消费者问题**：
- 生产者线程生成数据并放入缓冲区
- 消费者线程从缓冲区取出数据并处理
- 需要保证缓冲区满时生产者等待，缓冲区空时消费者等待

**用条件变量实现：**

```c
pthread_mutex_t mutex;
pthread_cond_t not_full;
pthread_cond_t not_empty;

// 生产者线程
void* producer(void* arg) {
    while (1) {
        pthread_mutex_lock(&mutex);
        while (buffer_is_full()) {
            pthread_cond_wait(&not_full, &mutex);
        }
        // 生产数据
        add_to_buffer(data);
        pthread_cond_signal(&not_empty);
        pthread_mutex_unlock(&mutex);
    }
}

// 消费者线程
void* consumer(void* arg) {
    while (1) {
        pthread_mutex_lock(&mutex);
        while (buffer_is_empty()) {
            pthread_cond_wait(&not_empty, &mutex);
        }
        // 消费数据
        data = remove_from_buffer();
        pthread_cond_signal(&not_full);
        pthread_mutex_unlock(&mutex);
    }
}
```

- **NSOperationQueue**：基于 GCD 的面向对象封装，提供更多控制选项

**4. 音视频同步的三种常见策略是什么？**

**答案：**

2. **视频向音频同步**：以音频时钟为基准，调整视频播放速度（最常用）
3. **向外部时钟同步**：以外部参考时钟为基准，同时调整音频和视频

**5. 在实际播放器开发中，为什么通常选择音频向视频同步或以音频为基准？**

**答案：**

- 音频卡顿或不同步会导致明显的听觉不适
- 视频轻微的不同步通常不易被察觉
- 音频设备的时钟通常更稳定，适合作为基准

**6. 如何计算当前音视频的播放时钟？**

**答案：**

**视频播放时钟计算：**
- 基于帧的 PTS（Presentation Time Stamp）
- 计算公式：`current_time = start_time + (current_frame_pts - first_frame_pts)`

**音频播放时钟计算：**
- 基于音频设备当前播放的采样位置
- 计算公式：`current_time = start_time + (current_sample_position / sample_rate)`

**7. 丢帧策略是如何设计的？**

**答案：**
丢帧策略设计：

1. **判断是否需要丢帧**：比较视频帧的 PTS 与当前音频时钟
2. **丢帧条件**：当视频帧的 PTS 落后于音频时钟超过一定阈值（如 100ms）时
3. **丢帧选择**：优先丢弃 B 帧，然后是 P 帧，尽量保留 I 帧
4. **丢帧数量**：根据时间差决定丢弃的帧数，避免一次性丢弃过多导致画面跳变
5. **恢复策略**：当视频追上音频时钟后，恢复正常播放

**8. 在解码线程和渲染线程之间，如何设计一个安全的音视频帧缓冲区？**

**答案：**
设计安全的音视频帧缓冲区：

2. **引用计数**：对帧数据使用引用计数，避免野指针
3. **最大容量**：设置队列最大容量，防止内存爆炸

**6. 如何用单链表实现一个简单的 FIFO 队列？**

**答案：**
```c
// 节点结构体
typedef struct Node {
    void* data;
    struct Node* next;
} Node;

// 队列结构体
typedef struct {
    Node* head;
    Node* tail;
    int size;
} Queue;

// 初始化队列
void queue_init(Queue* queue) {
    queue->head = NULL;
    queue->tail = NULL;
    queue->size = 0;
}

// 入队
void queue_push(Queue* queue, void* data) {
    Node* new_node = (Node*)malloc(sizeof(Node));
    new_node->data = data;
    new_node->next = NULL;
    
    if (queue->tail == NULL) {
        queue->head = new_node;
        queue->tail = new_node;
    } else {
        queue->tail->next = new_node;
        queue->tail = new_node;
    }
    queue->size++;
}

// 出队
void* queue_pop(Queue* queue) {
    if (queue->head == NULL) {
        return NULL;
    }
    
    Node* temp = queue->head;
    void* data = temp->data;
    
    queue->head = queue->head->next;
    if (queue->head == NULL) {
        queue->tail = NULL;
    }
    
    free(temp);
    queue->size--;
    return data;
}

// 销毁队列
void queue_destroy(Queue* queue) {
    while (queue->head != NULL) {
        Node* temp = queue->head;
        queue->head = queue->head->next;
        free(temp);
    }
    queue->tail = NULL;
    queue->size = 0;
}
```

**7. 在 C 中实现一个引用计数的数据结构，需要考虑哪些问题？**

**答案：**
需要考虑的问题：
- **线程安全**：引用计数的增减操作需要是原子的
- **内存管理**：当引用计数降为 0 时，正确释放资源
- **循环引用**：避免循环引用导致内存泄漏
- **接口设计**：提供清晰的 retain 和 release 接口

**示例：**
```c
typedef struct {
    int ref_count;
    // 其他数据成员
    pthread_mutex_t mutex; // 用于线程安全
} RefCounted;

// 初始化
void ref_counted_init(RefCounted* obj) {
    obj->ref_count = 1;
    pthread_mutex_init(&obj->mutex, NULL);
}

// 增加引用计数
void ref_counted_retain(RefCounted* obj) {
    pthread_mutex_lock(&obj->mutex);
    obj->ref_count++;
    pthread_mutex_unlock(&obj->mutex);
}

// 减少引用计数，当为 0 时释放
void ref_counted_release(RefCounted* obj) {
    pthread_mutex_lock(&obj->mutex);
    obj->ref_count--;
    int count = obj->ref_count;
    pthread_mutex_unlock(&obj->mutex);
    
    if (count == 0) {
        // 释放资源
        pthread_mutex_destroy(&obj->mutex);
        free(obj);
    }
}
```

### 1. 如何构建一个从零开始的 RTSP 播放器架构？

### 2. 解决 RTP over UDP 传输时的抖动和乱序问题

**解决方案：**

#### 引入 Jitter Buffer（抖动缓冲区）
- **位置**：在 RTP 接收模块之后，解码模块之前
- **作用**：存储一定数量的 RTP 包，根据时间戳和序列号进行排序和重传

#### 排序
- **实现**：根据 RTP 包的 `sequence number` 对乱序的包进行重排
- **算法**：使用优先级队列或环形缓冲区，按序列号排序

#### 抗抖动
- **缓冲区大小**：通常设计为可动态调整（如 200ms - 2000ms）
- **动态调整策略**：
  - 当网络抖动较小时，减小缓冲区大小，降低延迟
  - 当网络抖动较大时，增大缓冲区大小，提高稳定性

#### 丢包处理

**视频丢包处理：**
- **关键帧丢失**：通过 RTCP FIR（Full Intra Request）或 PLI（Picture Loss Indication）请求服务器重新发送关键帧
- **非关键帧丢失**：
  - 使用错误隐藏技术（如帧内预测、运动补偿）
  - 直接等待下一个关键帧到来，显示上一帧或黑屏

**音频丢包处理：**
- **PLC（Packet Loss Concealment）**：根据前后音频数据插值模拟出丢失的音频包
- **静音处理**：如果无法模拟，插入短暂的静音

#### 实现示例

```c
// Jitter Buffer 结构体
typedef struct {
    RtpPacket* packets[MAX_BUFFER_SIZE];
    int head;
    int tail;
    int size;
    int max_size;
    int64_t last_seq;
    pthread_mutex_t mutex;
} JitterBuffer;

// 添加 RTP 包到 Jitter Buffer
int jitter_buffer_add(JitterBuffer* jb, RtpPacket* packet) {
    pthread_mutex_lock(&jb->mutex);
    
    // 检查是否是重复包
    if (packet->seq <= jb->last_seq) {
        pthread_mutex_unlock(&jb->mutex);
        return -1;
    }
    
    // 检查缓冲区是否满
    if (jb->size >= jb->max_size) {
        // 丢弃最旧的包
        free(jb->packets[jb->head]);
        jb->head = (jb->head + 1) % jb->max_size;
        jb->size--;
    }
    
    // 按序列号插入到正确位置
    int insert_pos = jb->tail;
    while (insert_pos != jb->head && 
           jb->packets[(insert_pos - 1 + jb->max_size) % jb->max_size]->seq > packet->seq) {
        jb->packets[insert_pos] = jb->packets[(insert_pos - 1 + jb->max_size) % jb->max_size];
        insert_pos = (insert_pos - 1 + jb->max_size) % jb->max_size;
    }
    
    jb->packets[insert_pos] = packet;
    jb->tail = (jb->tail + 1) % jb->max_size;
    jb->size++;
    jb->last_seq = packet->seq;
    
    pthread_mutex_unlock(&jb->mutex);
    return 0;
}

// 从 Jitter Buffer 取出包
RtpPacket* jitter_buffer_get(JitterBuffer* jb) {
    pthread_mutex_lock(&jb->mutex);
    
    if (jb->size == 0) {
        pthread_mutex_unlock(&jb->mutex);
        return NULL;
    }
    
    RtpPacket* packet = jb->packets[jb->head];
    jb->head = (jb->head + 1) % jb->max_size;
    jb->size--;
    
    pthread_mutex_unlock(&jb->mutex);
    return packet;
}
```

### 3. iOS 平台上的最佳实践：如何避免内存爆炸和卡顿

#### 内存管理

**1. 控制队列大小**
- **实现**：为音视频待渲染队列设置最大容量
- **策略**：当网络卡顿时，解码线程不能无限生产，当队列满时：
  - 解码线程应等待
  - 或主动丢弃旧的未解码包

**2. 重用对象**
- **实现**：使用 `av_frame_alloc` 预先分配好 `AVFrame`，解码后填充，渲染后 `av_frame_unref` 重置
- **优势**：避免频繁 `alloc` 和 `free` 导致的内存碎片

**3. 使用内存池**
- **实现**：对于频繁申请的临时缓冲区（如 RTP 接收缓冲区），实现一个简单的内存池
- **优势**：减少内存分配开销，提高性能

**4. 监控内存使用**
- **实现**：定期检查应用内存使用情况
- **策略**：当内存使用接近上限时，主动释放一些非必要资源

#### 性能优化

**1. 硬解码**
- **实现**：优先使用 `VideoToolbox` 进行硬解码
- **优势**：极大降低 CPU 占用，减少发热

**2. 零拷贝渲染**
- **实现**：利用 `CVOpenGLESTextureCache` 直接渲染 `CVPixelBufferRef`
- **优势**：避免内存拷贝，提高渲染效率

**3. 线程优先级**
- **实现**：适当提高音频渲染线程的优先级
- **优势**：确保声音输出的连续性，避免音频卡顿

**4. 异步处理**
- **实现**：所有可能阻塞主线程的操作（如网络请求、解码）都必须放在子线程
- **优势**：保证 UI 响应流畅

**5. 帧率控制**
- **实现**：根据设备性能和网络状况，动态调整视频帧率
- **优势**：在低端设备上也能保持流畅播放

**6. 缓存策略**
- **实现**：合理设置缓存大小，平衡延迟和流畅度
- **策略**：
  - 网络好时，减小缓存，降低延迟
  - 网络差时，增大缓存，提高稳定性
